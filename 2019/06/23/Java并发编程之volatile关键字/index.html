Java并发编程之volatile关键字
发表于 2019-06-23| 更新于 2019-10-08|Java
字数总计: 2.5k|阅读时长: 7 分钟|阅读量:

前言

由于最近有这么一个业务需要在多个线程中对同一个变量进行加减操作. 考虑到如果使用synchronized对值操作进行同步化是否过于影响性能. 之前听说过这么一句话: volatile可以说是Java虚拟机提供的最轻量级的同步化机制. 误以为可以使用该关键字更加优雅地解决问题.

结论：通过深入学习后, 了解到volatile并不适用于我提到的业务场景.
引用《深入理解Java虚拟机》中的结论:

在不符合一下两条规则的运算场景中，我们仍然要通过加锁(使用synchronized或java.util.concurrent中的原子类)来保证原子性.

  • 运算结果并不依赖变量当前的值, 或者能够确保只有单一的线程修改变量的值.
  • 变量不需要与其他的状态变量共同参与不变约束.

也就是volatile仅仅适用于上述两个场景.

尽管没有很好地解决我想要解决的问题, 但还这次的学习还是有所意义. 接下来便与大家讨论一下关于volatile关键字.

概述

上文中我提到了这么一句话: volatile可以说是Java虚拟机提供的最轻量级的同步化机制.
实际上并没有错, 虽然没有解决我的问题, 但是它确实是轻量级的同步化机制的体现. 那么我们来看看它能够提供的机制是什么?
其实它有两层语义, 以耽一个共享变量被volatile修饰, 他就具备一下两个特性:

  1. 保证了不同线程对这个变量进行操作时的可见性, 即一个线程修改了某个变量的值, 这个新值对其他线程来说是立即可见的.
  2. 禁止进行指令重排序

看到这俩特性大概是感觉晦涩难懂, 甚至没听说过什么可见性, 重排序啥玩意. 所以接下来进行详细介绍.

volatile具体说明

JMM概述

在了解可见性的具体意义之前, 我们先聊聊Java的内存模型, 也就是所谓的JMM(Java Memory Model), 其实JMM的设计与我们物理机(也就是我们的计算机)中的内存设计相近, 所以我想通过物理机来解释一下JMM, 相信这样大家会更易于接受一些.
物理机的内存模型(处理器与寄存器间交互概述)
接触过计算机的大家都知道处理器, 也知道处理器的运算速度远大于(几个数量级的差距)计算机中的存储设备(比如内存), 所以我们都愿意通过处理器来计算数据, 但不可能把数据全都存于处理器中, 所以现代计算机都不得不及射入一层读写速度解决处理器运算的高速缓存区作为内存与处理器中间的缓冲:

  1. 将运算需要用到的数据赋值到缓存中, 让运算能够快速进行
  2. 当运算结束后再从缓存同步回到内存中, 这样处理器就无须等待缓慢的内存读写了

(图源于《深入理解java虚拟机》)

JMM
java的内存模型是这样的: 有一个共享的主内存, 每个线程有各自的工作内存. 需要运算的数据从主内存中取出, 运算结束后将结果同步回主内存中.

(图源于《深入理解java虚拟机》)

可见性

了解了上述的JMM后, 大家可以继续对可见性进行了解了, 其实各线程的工作内存是互相不可见的.

其实在内存角度分析该代码, 应该是

  1. 主内存中有一个值为1的变量i
  2. 线程1将主内存中的i读取到其工作内存中, 将该i自增
  3. 线程2将主内存中的i读取到其工作内存中, 将该i自增
    线程1与线程2中的变量i是互不可见的, 可以这么理解, 线程1中的i自增了, 但尚未将修改后的值save和load到主内存之前, 线程2是看不到i的改变的.

而使用了volatile关键字修饰i的话, 线程1和线程2对这个变量i的修改是可见的.

我之所以以为这样可以用于我们多线程进行变量修改的场景, 就是因为这个可见性让我误以为并发修改的问题主要源于这个内存模型导致不可见的问题. 实际上还有原子性问题, volatile只能保证可见性, 不能保证原子性.

这里面就有一个误区了, volatile关键字能保证可见性没有错, 但是上面的程序错在没能保证原子性. 可见性只能保证每次读取的是最新的值, 但是volatile没办法保证对变量的操作的原子性.

指令重排

volatile的语义的第二条为: 禁止进行指令重排序
什么叫做指令重排序呢?
其实在java内存中有一个这样的描述: 线程内表现为串行的语义
其意义是普通的变量仅仅保证在执行过程中, 所有依赖赋值结果的地方能够获取到正确结果, 但是不能保证变量赋值操作的顺序与代码中执行顺序一致.
简言之就是代码的执行顺序不一定与你的书写顺序一致. 只要没有依赖关系, 是不一定会按顺序执行代码的.
