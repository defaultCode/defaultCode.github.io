Java并发包之Atomic原子操作类
发表于 2019-07-01| 更新于 2019-10-08|Java
字数总计: 2.5k|阅读时长: 8 分钟|阅读量:

引言

在上一篇文章讨论volatile关键字时提及了Java的原子操作类. 今天我们就来对这些原子操作类进行学习.

由于我们对变量的更新操作多为加减乘除, 这些操作不是原子的, 所以我们使用volatile并不能很好地解决高并发环境下变量进行加减操作时产生非预期值的问题(volatile仅仅可以保证变量的可见性). 通常我们使用synchronized来解决该问题, 但众所周知, synchronized的性能非常差, 而且只能用在方法或块上, 而如果只是更新一个变量使用一个代码块来包裹, 让人总有一种牛刀杀鸡的感觉. 或许Java开发团队也这么觉得吧, 于是就有了原子操作类.

以下为《java并发编程的艺术》一书中对原子操作类的描述.

而Java从JDK1.5开始提供了java.util.concurrent.atomic包(也就是JUC中的Atomic包, 接下来会简称为Atomic包), 这个包中的原子操作类提供了一种用法简单, 性能高效, 线程安全的方式给我们用于更新变量.

至于看到这个描述, 似乎让人难以拒绝, 既能保证线程安全, 还简单, 又高效. 接下来我们深入了解, 看看是否有想象中的简单, 高效.

是否高效?

我们先用一段代码测试一下这个原子操作类和synchronized的效率差距.

通过修改THREADS的值, 在我的笔记本中运行得到的数据如下:

线程数 1 10 100 1000 10000
atomic 958ms 2508ms 2548ms 2528ms 2553ms
synchronized 2828ms 4436ms 4479ms 4411ms 5450ms

我们可以看到atomic(包中类的运算方法)的性能确实优于synchronized, 但其实线程增加到一定程度时, 已经不会将性能拉差距得更大.

原子操作类有哪些?

我们先看看JDK8中有哪些原子操作类

原子更新基本类型类

使用原子的方式更新基本类型, Atomic包提供了AtomicBoolean, AtomicInteger, AtomicLong这3个类. 相信看类名大家也都知道是对哪些基本类型进行操作了, 由于这个类都极为相似, 我们通过AtomicInteger来看看这些类是这么用的.
我这里就不一一将AtomicInteger类的所有方法详述了, 有兴趣的同学可以看看这个类的其他API, 我们先看看常用的方法:

  • int addAndGet(int delta): 以原子的方式将输入的数值与实例中的值相加, 并返回结果.
  • boolean compareAndSet(int expect, int update): 如果输入的值等于预期值, 则以原子方式将该值设置为输入的值.
  • int getAndIncrement(): 以原子的方式将当前值加 1, 注意, 这里返回的是自增前的值, 也就是旧值(同i++).
  • void lazySet(int newValue): 最终会设置成newValue, 使用lazySet设置值后, 可能导致其他线程在之后的一小段时间内还是可以读到旧的值.
  • int getAndSet(int newValue): 以原子的方式设置为newValue, 并返回旧值.
    这些都是类调用方法这么简单的事, 相信大家看描述也知道一些方法如何使用以及应用场景, 对有一些方法, 应该会有所疑惑, 不管你有没有疑惑, 我反正有疑惑, 所以特地做了了解, 希望跟大家分享以下.
lazySet?

从描述上看, 这个方法会可能会导致其他线程在之后的一小段时间内还是可以读到旧的值.
一开始看到这段描述的时候, 我觉得很奇怪, 这样岂不是将可见性这点给去除了? 有什么意义吗?

通过查阅了相关资料后, 了解到可见性是通过进行设置内存屏障来保证的, 而这一方法消除了一些内存屏障来保证更高的性能, 实际上也就是与volatile关键字恰恰相反, 它能够保证原子性, 不保证可见性.

原子更新数组

顾名思义, 通过原子的方式更新数组中的某个元素. Atomic包提供了AtomicIntegerArray, AtomicLongArray,AtomicReferenceArray这3个类.
主要使用两个方法:

  • get(int index): 获取索引为index的元素值.
  • compareAndSet(int i, int expect, int update): 如果当前值等于预期值, 则以原子方式将数组位置 i 的元素设置为update值.

这些方法用法非常简单, 就不过多解释了.

原子更新引用类型

原子地更新对象使用的类, Atomic包提供了3个类:

  • AtomicReference: 原子更新引用类型
  • AtomicReferenceFieldUpdater: 原子更新引用类型的字段
  • AtomicMarkableReferce: 原子更新带有标记位的引用类型. 可以使用构造方法更新一个布尔类型的标记位和引用类型

相信大家和我一样, 前两个可以理解, 第三个不明白是啥玩意. 所以我还是在这里给大家说明一下:

可能认真的同学已经打开上面的图看到了一个这样的类: AtomicStampedReference这个类记录了引用的时间戳, 保证时间戳的唯一就可以解决CAS中的ABA问题. 而AtomicMarkableReferce是以标记是否被修改过形式(用一个Boolean来维护)来解决该问题的.

什么是CAS?

java.util.concurrent包中借助CAS实现了区别于synchronouse同步锁的一种乐观锁。
CAS有3个操作数，内存值V，旧的预期值A，要修改的新值B。当且仅当预期值A和内存值V相同时，将内存值V修改为B，否则什么都不做。

什么是ABA?

我们先来看一个多线程的运行场景：
时间点1 ：线程1查询值是否为A
时间点2 ：线程2查询值是否为A
时间点3 ：线程2比较并更新值为B
时间点4 ：线程2查询值是否为B
时间点5 ：线程2比较并更新值为A
时间点6 ：线程1比较并更新值为C
在这个线程执行场景中，2个线程交替执行。线程1在时间点6的时候依然能够正常的进行CAS操作，尽管在时间点2到时间点6期间已经发生一些意想不到的变化， 但是线程1对这些变化却一无所知，因为对线程1来说A的确还在。通常将这类现象称为ABA问题。
ABA发生了，但线程不知道。又或者链表的头在变化了两次后恢复了原值，但是不代表链表就没有变化。

原子更新字段类

如果需要原子的更新类里某个字段时, 需要用到原子更新字段类, Atomic包提供了3个类进行原子字段更新:

  • AtomicIntegerFieldUpdater: 原子更新整型的字段的更新器
  • AtomicLongFieldUpdater: 原子更新长整型字段的更新器
  • AtomicStampedFieldUpdater: 原子更新带有版本号(时间戳)的引用类型

JDK8中原子类新成员

DoubleAccumulator,LongAccumulator, DoubleAdder, LongAdder. 这几个类为JDK8中新加入的原子类.
这些类对应把 AtomicLong 等类的改进。比如 LongAccumulator与 LongAdder 在高并发环境下比 AtomicLong 更高效。
我个人认为只要大概了解

  • Atomic、Adder在低并发环境下, 两者性能很相似. 但在高并发环境下, Adder 有着明显更高的吞吐量, 但是有着更高的空间复杂度.
  • sum() 方法在没有并发的情况下调用, 如果在并发情况下使用会存在计数不准.

顺便一说《阿里巴巴开发手册》中提出:

【参考】volatile 解决多线程内存不可见问题。对于一写多读，是可以解决变量同步问题， 但是如果多写，同样无法解决线程安全问题。如果是 count++操作，使用如下类实现： AtomicInteger count = new AtomicInteger(); count.addAndGet(1); 如果是 JDK8，推 荐使用 LongAdder 对象，比 AtomicLong 性能更好（减少乐观锁的重试次数）。
